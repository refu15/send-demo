## 0. 前提

この要件定義は、実装前の研究・設計フェーズの成果物である。ユーザー承認前にアプリ本体のコードは書かない。

現時点の前提:

- Webアプリとして作る
- 初回デモはローカルPCで動かす
- 画面言語は日本語
- 送迎順はExcelの順番を正とする
- 初回デモでは地図APIを使わない
- 初回デモではGPS常時取得をしない
- 初回デモではログイン認証を作らず、簡易ロール切替にする
- 実Excelには個人情報が含まれるため、外部APIへ送らない
- 実ExcelファイルはGitにコミットしない

この前提を変える場合は、実装前にこの要件定義を更新する。

## 1. ゴール

アプリ内で利用者、車両、職員、送迎計画、運行実績を一元管理できる送迎支援デモシステムを作る。

参考にする体験は、ダイハツの送迎支援システム「らくぴた送迎」のような、送迎前、送迎中、送迎後をつなぐ業務フローである。Excelは業務の正ではなく、外部提出・紙配布・バックアップ用のエクスポート形式として扱う。

デモの成功状態は、以下を第三者に画面で説明できること。

- アプリ内で当日の送迎計画を作成・編集できる
- 管理者が車両別、AM/PM別の送迎予定を見られる
- ドライバーがスマホ想定画面で到着、乗車/降車、出発、完了を記録できる
- 施設側が送迎進捗を一覧で確認できる
- 送迎後に予定時刻と実績時刻を比較し、Excel/CSVへ出力できる

## 6. 利用者ロール

### 管理者

目的:

送迎計画を取り込み、当日の配車・担当者・進捗を管理する。

できること:

- 利用者、車両、職員を登録・編集する
- 送迎計画を作成・編集する
- 送迎計画を確認する
- 車両、運転手、添乗員を確認する
- 利用者の順番を確認する
- 欠席、キャンセルを登録する
- 運行状況を見る
- 送迎表と実績をExcel/CSVで出力する

### 施設職員

目的:

送迎の進み具合と遅れを把握し、利用者・家族からの問い合わせに答える。

できること:

- 車両別の進捗を見る
- 未出発、送迎中、完了、キャンセルを見る
- 遅れのある便を確認する
- 備考を確認する

### ドライバー・添乗員

目的:

今日の送迎順と訪問先を確認し、停車中に実績を記録する。

できること:

- 自分の担当便を見る
- 次の訪問先を見る
- 住所、電話、備考を見る
- 到着を記録する
- 乗車または降車を記録する
- 出発を記録する
- 完了を記録する
- キャンセルを記録する

## 7. 業務フロー

### 送迎前

1. 管理者が利用者、車両、職員マスタを登録する
2. 管理者が当日のAM便、PM便を作成する
3. 車両、運転手、添乗員を割り当てる
4. 利用者を送迎順に並べる
5. 当日のAM便、PM便を確定する
6. 必要に応じて送迎表Excelを出力する
7. ドライバーに担当便が表示される

### 送迎中

1. ドライバーがスマホ画面で担当便を開く
2. 次の利用者を確認する
3. 到着時に「到着」を押す
4. 乗車または降車時に「乗車/降車」を押す
5. 出発時に「出発」を押す
6. 施設側の進捗画面に状態が反映される
7. キャンセルや遅れがあれば記録する

### 送迎後

1. 管理者が実績画面を開く
2. 予定時刻と実績時刻を比較する
3. 遅延、キャンセル、備考を確認する
4. CSVを出力する
5. 必要に応じて次回の送迎順見直しに使う

## 8. 機能要件

### F-001 マスタ管理

利用者、車両、職員をアプリ内で登録・編集・削除できる。

受入条件:

- 利用者の氏名、利用曜日、住所、電話、車椅子、薬確認、備考を登録できる
- 車両名、定員、備考を登録できる

### F-001b 既存Excel初期移行

既存Excelを初回だけ読み取り、アプリ内DBへデータ移行できる。通常運用では使わない。

受入条件:

- 既存Excelから利用者、車両、職員、当日便候補を作成できる
- 移行結果を確認してから保存できる
- 移行後はアプリ内データを正とする

受入条件:

- 車両別の進捗が表示される
- 未出発、送迎中、完了、キャンセル件数が表示される
- 遅延候補が表示される

### F-003 配車計画画面

アプリ内で作成した計画を車両別、送迎順で表示・編集する。

受入条件:

スマホ幅で使える運行画面を提供する。

受入条件:

施設側が車両別の運行状況を確認できる。

受入条件:

- 各車両の現在ステータスが見える
- 何人完了し、何人残っているか見える

### F-006 実績画面

送迎後に予定と実績を比較できる。

受入条件:

- 利用者ごとに予定時刻、到着時刻、出発時刻、完了時刻が見える

### F-007 簡易ロール切替

デモ用に管理者、施設職員、ドライバーを切り替えられる。

受入条件:

## 11. 状態遷移

RideStopの状態:

- planned: 予定
- skipped: キャンセル
- arrived: 到着
- boarded: 乗車または降車
- departed: 出発
- completed: 完了

基本遷移:

`planned -> arrived -> boarded -> departed -> completed`

例外遷移:

`planned -> skipped`

取り消し:

- 直前のイベントを1つだけ取り消せる
- 取り消し履歴は残す

## 13.1 コードスタイル

方針:

- TypeScriptは型を明示する
- Excel取込処理はUIから分離する
- 送迎状態の更新は純粋関数としてテストできる形にする
- 画面表示用の文言と業務ロジックを混ぜない
- 個人情報をconsole出力しない
- 例外は握りつぶさず、取込エラーとして表示できる形にする

状態遷移関数の例:

```ts
type RideStopStatus =
  | "planned"
  | "skipped"
  | "arrived"
  | "boarded"
  | "departed"
  | "completed";

type RideEventType =
  | "skip"
  | "arrive"
  | "board"
  | "depart"
  | "complete";

export function nextRideStopStatus(
  current: RideStopStatus,
  eventType: RideEventType,
): RideStopStatus {
  const transitions: Record<RideStopStatus, Partial<Record<RideEventType, RideStopStatus>>> = {
    planned: { skip: "skipped", arrive: "arrived" },
    skipped: {},
    arrived: { board: "boarded" },
    boarded: { depart: "departed" },
    departed: { complete: "completed" },
    completed: {},
  };

  const next = transitions[current][eventType];

  if (!next) {
    throw new Error(`Invalid ride stop transition: ${current} -> ${eventType}`);
  }

  return next;
}
```

## 16. リスクと対策

### リスク1: CRUDが増えて初回開発範囲が膨らむ

対策:

- 最初は利用者、車両、職員、送迎計画の最小項目に絞る
- 認証、本番権限、複数事業所は後回しにする
- Excel取込は初期移行機能に下げる

### リスク2: 個人情報の扱い

対策:

- デモはローカル実行に限定する
- ExcelファイルをGitに入れない
- 画面共有時は必要最小限の情報だけ出す
- 本番化時に認証、権限、監査ログを入れる

### リスク3: 最適化への期待が膨らむ

対策:

- デモではアプリ内で設定した送迎順を尊重する
- ルート最適化は第2段階にする
- 地図APIと住所正規化が必要なことを明示する

### リスク4: ドライバー操作が危険になる

対策:

- 停車中のワンタップ操作だけにする
- 操作ボタンを大きくする
- 運転中操作を促す文言を避ける

## 17. 境界条件

### Always

- テストが通るまでコミットしない
- コミット前に `git diff` を確認する
- Excel実データをGitに入れない
- 個人情報を外部サービスに送らない
- 実装前に要件を更新する
- アプリ内DBをデータの正とする

### Ask First

- 外部地図APIを使う
- SMS、LINE、電話通知を入れる
- 認証方式を決める
- DBスキーマを本番前提に変える
- 実Excelファイルをリポジトリにコピーする
- Excel取込を通常運用フローに戻す

### Never

- OAuthトークン、APIキー、個人情報をコミットする
- テスト失敗のままコミットする
- 実データを外部AI/APIへ送る
- ユーザー承認なしに本番運用前提の設計へ広げる

5. デモ中に実名、住所、電話番号を表示してよいか
6. DB保存はSQLiteでよいか

## 19. 初回デモの推奨判断

初回デモでは以下を選ぶ。

- ローカルPCで動かす
- 日本語のみ
- 地図なし
- Excelはエクスポートのみ
- 実名、住所、電話番号はマスク表示も選べるようにする
- SQLiteに保存する

理由:

- 送迎業務の価値を最短で見せられる
- 外部API契約なしで作れる
- 個人情報を外に出さずに済む
- 将来の本番化に向けた構造を残せる

## 20. 開発着手ゲート

開発に進む前に、ユーザーが次の点を理解して承認する必要がある。

- 初回デモは「自動最適配車」ではなく「アプリ内での送迎管理、運行進捗、実績記録、Excel/CSV出力」である
- GPS、通知、地図、AI最適化は初回範囲外である
- Excelは入力元ではなく出力形式である
- デモ完成の検証は、ローカル起動、マスタ登録、送迎計画作成、画面操作、Excel/CSV出力、テスト通過で行う

承認前の理解確認:

1. 初回デモの中心機能は何か
2. 初回デモで意図的に作らない機能は何か
3. 実Excelの個人情報をどう扱うか
4. 完了判定は何で行うか
