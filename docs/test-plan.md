## 0. 現方針

このシステムでは、アプリ内DBをデータの正とする。Excelは通常運用の入力元ではなく、送迎表・実績表を外部へ渡すためのエクスポート形式である。

したがって、テストの最優先対象は次の順にする。

1. DBモデルとCRUD
2. 利用者、車両、職員のマスタ管理
3. AM/PM便と送迎順の作成・編集
4. ドライバー操作と状態遷移
5. 進捗・実績の集計
6. 送迎表Excel出力
7. 実績CSV出力
8. 既存Excelからの初期移行

既存Excel取込テストは、通常運用ではなく初期移行機能のテストとして扱う。

## 1. 目的

初回デモが「動いて見える」だけでなく、送迎業務として壊れてはいけない部分をテストで証明できる状態にする。

重点は以下の5点。

- アプリ内DBでマスタと送迎計画を管理できる
- 送迎状態遷移が破綻しない
- 管理者、施設職員、ドライバーの主要画面が成立する
- 送迎表Excel出力と実績CSV出力が検証できる
- 個人情報を不用意に外部送信、ログ出力、コミットしない

## 2. テスト方針

### 優先順位

1. Unit test: 業務ロジック、DB repository、状態遷移、Excel/CSV出力
2. Integration test: マスタ登録から送迎計画作成、運行実績まで
3. Component test: 重要画面部品の表示と操作
4. E2E test: 初回デモの主要導線
5. Manual acceptance: 実Excelを使った受入確認

### テストピラミッド

- Unit: 70%
- Integration: 20%
- E2E / Manual: 10%

### 初回デモで必須の合格条件

- `npm run lint` が通る
- `npm run test -- --run` が通る
- `npm run build` が通る
- 手動デモチェックリストが通る
- `docs/ux-ui-design.md` のUI実装チェックリストが通る
- `git diff` で差分を確認済み
- 実ExcelファイルがGit管理対象に入っていない

## 3. テスト環境

### ローカル

- Windows
- Node.js
- Next.js
- TypeScript
- Vitest
- Playwrightは必要になった時点で導入

### テストデータ

使用するデータ:

- 実ファイル: `C:\Users\nnkre\Downloads\up1送迎表.xlsm`
- テスト用fixture: 実Excelから個人情報を除いた最小サンプル
- 手動確認: アプリ内登録、送迎計画作成、Excel/CSV出力

禁止:

- 実ExcelをGitにコミットする
- 実名、住所、電話番号をテストスナップショットに残す
- 実データを外部APIへ送る

## 4. テスト分類

| 分類 | 対象 | 主な検証 | コマンド |
| --- | --- | --- | --- |
| Unit | 純粋関数 | 状態遷移、集計、Excel/CSV出力、マスク | `npm run test -- --run` |
| Integration | DB/CRUD | マスタ登録、送迎計画作成、実績イベント | `npm run test -- --run` |
| Component | UI部品 | 表示、ボタン、ロール切替 | `npm run test -- --run` |
| E2E | 主要導線 | 取込から実績出力まで | `npm run test:e2e` |
| Manual | 実Excel | 操作感、画面崩れ、業務説明 | 手動 |
| Security | 個人情報 | ログ、Git、外部送信なし | 手動 + grep |
| Performance | デモ性能 | 取込5秒目標、表示即時 | 手動 + 計測 |

## 9. E2E Test

手順:

1. ローカルURLを開く
2. Excel取込画面を開く
3. `.xlsm` を選択する
4. 取込結果を確認する

## 16. 開発時のTDDルール

- 新しい業務ロジックは、先にテストを書く
- バグ修正は、先に再現テストを書く
- Excel取込の仕様変更は、fixtureと期待値を更新してから実装する
- 状態遷移の変更は、正常系と異常系の両方を追加する
- CSV列変更は、ヘッダーとエスケープのテストを更新する
- テストをskipして完了扱いにしない
- テストが通るまでコミットしない
