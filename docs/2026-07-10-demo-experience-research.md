## 見つかった課題

1. 初期表示が `マスタ管理` で、顧客が最初に見る価値が入力フォームと長い一覧になっている。
2. スマホではヘッダーとナビは押せるが、初期画面の情報量が多く、体験導線が弱い。
3. PCでも初期表示が編集業務中心で、送迎計画、運行、進捗、実績の流れがすぐ伝わらない。
4. `favicon.ico` が 404。
5. パスワードフォームに username フィールドなしのブラウザ警告が出る。
6. `電話` ボタンは見た目上は実操作に見えるが、実際の発信・連携は未実装。デモ上の誤解リスクがある。

## 推奨スコープ

### REQ-DEMO-001: 初回体験の改善

要求:
静的デモ初回表示を `今日の運行` から始め、現場が一日の送迎業務を上から順番に確認できるようにする。

受入条件:
- パスワード通過後、スマホ/PCともに入力フォームではなく現場オペレーションの順番が最初に見える。
- 現場データ、配車計画、ドライバー操作、進捗、実績、Excel出力への導線が迷わない。

### REQ-DEMO-002: スマホ/PCナビゲーションの仕上げ

要求:
スマホでは主要導線を押しやすく、PCでは横並びでも詰まらないようにする。

受入条件:
- 390px幅で主要ボタンのテキストがはみ出さない。
- 1440px幅でヘッダー、ナビ、本文が過剰に散らばらない。
- ドライバー画面は高齢者でも押し間違えにくい大きさと文言にする。

### REQ-DEMO-003: 公開デモ品質の軽微改善

要求:
デモ公開時のコンソールノイズと誤解を減らす。

受入条件:
- `favicon.ico` 404 が出ない。
- パスワードフォームのブラウザ警告を抑える、または許容理由を明記する。
- 未実装の `電話` 操作が本番機能と誤解されない。

## 対象外

- 本番認証、監査ログ、権限管理の実装
- GPS、地図、ルート最適化
- SMS、LINE、電話通知など外部連携
- 実データ・個人情報の投入
- GitHub Pages 以外の本番環境設計
- DBスキーマの大幅変更

## Devil's Advocate

### Critical

- 初期表示だけを変えても、デモの成功条件が「何を見れば合格か」になっていなければ顧客確認が散る。画面改修と一緒に確認手順を明示する必要がある。
- GitHub Pages のパスワードゲートは静的な簡易ゲートであり、機密データを扱えるものではない。公開デモのデータは必ず架空データに限定する。

### Warning

- スマホ最適化をしすぎると、PCでの管理者向け情報密度が落ちる。ドライバー画面はスマホ優先、管理・進捗画面はPCでも読みやすくする切り分けが必要。
- テーブルをカード化しすぎると既存の編集機能・テスト影響が大きい。今回の範囲では初期導線とナビ改善を優先する。

### Suggestion

- 最小実装案: 初期 view を `operations` に変更し、既存画面を現場オペレーション順につなぐ。
- 中間案: 初期 view を `dashboard` にし、ダッシュボードからドライバー・進捗・実績へ進める導線を追加する。
- 大きい案: 顧客向け専用の `デモツアー` 画面を新設する。

## 推奨案

最小実装案を採用した。

理由:
- 既存構造への影響が小さい。
- 静的デモでは初期データが自動投入済みなので、`今日の運行` から始めると朝の確認、運行、進捗、実績、出力の順番を見せられる。
- テスト・ビルドへの影響が小さく、スマホ/PC確認まで短時間で閉じられる。

## 実装結果

- `今日の運行` 画面を追加し、初期表示にした。
- `朝の確認 -> 車で使う -> 施設で見る -> 終わった後 -> 記録を出す` の順に、既存機能へ遷移できるようにした。
- ドライバー画面の主要操作を `到着した`、`乗った / 降りた`、`出発する`、`完了した` に変更し、ボタンを大きくした。
- `電話` は未実装の発信機能と誤解されないよう `連絡先確認` に変更した。
- favicon を追加し、公開デモの 404 ノイズを解消した。
- パスワードフォームのブラウザ警告を抑える username helper を追加した。
