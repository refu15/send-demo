## 1. 設計目的

`up1送迎表.xlsm` を起点にした送迎支援デモを、現場で説明しやすく、迷わず操作できる業務UIとして設計する。

このUIは、見栄えの良い管理画面ではなく、送迎前、送迎中、送迎後の判断と記録を減らすための作業道具である。

初回デモのUXゴール:

- Excel送迎表をWeb上で見える化できる
- 管理者が当日の全体像を10秒以内に把握できる
- ドライバーが次に何をするか迷わない
- 施設職員が問い合わせに即答できる
- 送迎後に予定と実績の差分を説明できる
- 個人情報を必要以上に露出しない

## 3. 調査からの設計原則

### 原則1: 現場負荷を増やさない

介護現場では人手不足と業務負担が前提である。UIは「入力を増やす」のではなく、「Excelで既にある情報を再利用し、記録漏れを減らす」方向に設計する。

設計反映:

- Excel取込を起点にする
- 手入力はキャンセル理由、実績操作、備考追記に限定する
- 同じ情報を複数画面で入力させない

### 原則2: 送迎前・中・後を分断しない

参考サービスは、送迎前の計画、送迎中の進捗共有、送迎後の記録管理を一連の業務として扱っている。デモも同じ構造にする。

設計反映:

- 画面ナビゲーションを「取込」「計画」「運行」「進捗」「実績」にする
- 送迎中のイベントが進捗と実績へ自動反映されるように見せる

### 原則3: 安全確認を操作フローに埋め込む

送迎業務では乗降確認、残人数確認、完了確認が重要である。初回デモでは置き去り防止装置そのものは作らないが、UI上の安全確認ポイントは設計する。

設計反映:

- ドライバー画面に「残り人数」「未完了」「次の訪問先」を常時表示する
- 完了操作前に、該当利用者の状態を明確に表示する
- キャンセルと完了を色だけで区別しない

### 原則4: アクセシビリティを最初から設計条件にする

施設職員、ドライバー、管理者は年齢・端末・視力・利用環境がばらつく。コントラスト、文字サイズ、キーボード操作、色以外の表現を最初から満たす。

設計反映:

- 本文テキストは背景に対して4.5:1以上を目標にする
- 状態は色、ラベル、アイコンを併用する
- ボタンはスマホで押しやすい高さにする
- フォーカスリングを明確にする

## 4. UXコンセプト

名称:

**送迎レーン**

コンセプト:

車両ごとの送迎順を、工場のラインや運行表のような「レーン」として扱う。各レーンは、予定、現在、完了、遅延を一目で確認できる。

このデモの象徴的なUI:

- 車両ごとの横または縦レーン
- 各利用者を停留所ノードとして表示
- 現在位置は太い境界線とラベルで示す
- 完了は塗りつぶし、未完了は白地、キャンセルは斜線パターン

採用理由:

- 送迎順が業務の中心だから
- 地図がなくても進捗を説明できるから
- Excelの行構造から自然に変換できるから
- 管理者、施設職員、ドライバーの全員に意味が通じるから

## 5. 対象ユーザー

### 管理者・計画担当者

状況:

- 朝の限られた時間で送迎表を確認する
- 欠席、車両変更、担当者変更に対応する
- 送迎後に実績を確認する

主な不安:

- Excelの転記ミス
- 誰がどの車両を担当しているか分からない
- 遅延やキャンセルが職員間で共有されない

UI方針:

- 一覧性を優先
- 情報密度は高め
- 変更操作は慎重に、確認可能にする

### 施設職員

状況:

- 利用者や家族から到着時刻を聞かれる
- 施設内の受け入れ準備をする
- 送迎の遅れを把握したい

主な不安:

- 今どの車がどこまで進んだか分からない
- 残り人数が分からない
- 問い合わせに答えられない

UI方針:

- 進捗、残人数、最終更新を最上位に置く
- 詳細編集は不要
- 表示は読み取り中心

### ドライバー・添乗員

状況:

- スマホで今日の担当便を見る
- 停車中に最小操作で記録する
- ナビ、電話、備考確認を行う

主な不安:

- 次の訪問先を間違える
- 押すボタンを間違える
- 運転中に画面操作が必要になる

UI方針:

- 次の1件を最優先
- 操作は大きなボタン
- 一画面一判断
- 運転中操作を促さない

## 7. 画面別UX/UI設計

### 7.1 Excel取込画面

目的:

Excel送迎表を取り込み、取込結果が業務的に妥当か確認する。

主要要素:

- ファイル選択
- 取込ボタン
- 取込結果サマリー
- エラー・警告一覧
- デモ表示マスク切替

レイアウト:

```text
┌─────────────────────────────┐
│ Excel送迎表を取り込む          │
│ [ファイル選択] [取込]          │
├─────────────────────────────┤
│ 取込結果: 2車両 / 3職員 / 10名 / AM 1便 / PM 1便 │
├─────────────────────────────┤
│ 警告: 予定時刻なし 2件          │
│ エラー: なし                    │
└─────────────────────────────┘
```

UI方針:

- 成功/失敗だけでなく「何件取り込めたか」を見せる
- エラーは行・列・項目・理由を表示する
- 実名表示とマスク表示を切り替えられる

### 7.2 ダッシュボード

目的:

当日の送迎状況を10秒で把握する。

主要要素:

- 日付、曜日、天気
- AM/PM切替
- 車両別送迎レーン
- 未出発、送迎中、完了、キャンセル数
- 遅延候補

レイアウト:

```text
┌──────────────────────────────────────┐
│ 2026/06/28 日曜  天気: 晴れ   [AM] [PM] │
├──────────────────────────────────────┤
│ 全体: 未出発 8 / 送迎中 1 / 完了 2 / キャンセル 0 │
├──────────────────────────────────────┤
│ 車両A  運転手: ... 添乗: ...            │
│ 予定 ○──○──●──□──□  完了2 残3        │
├──────────────────────────────────────┤
│ 車両B  運転手: ... 添乗: ...            │
│ 予定 □──□──□                         │
└──────────────────────────────────────┘
```

UI方針:

- レーンを主役にする
- 画面上部はKPIではなく「状態の判断」に必要な情報に限定する
- 遅延候補は赤だけでなく「遅延候補」ラベルを付ける

### 7.3 配車計画画面

目的:

車両別、送迎順で計画内容を確認し、欠席やキャンセルを反映する。

主要要素:

- 車両タブ
- AM/PM切替
- 送迎順テーブル
- 車椅子、薬確認、備考
- キャンセル操作

テーブル列:

- 順
- 予定
- 利用者
- 状態
- 車椅子
- 薬
- 備考
- 操作

UI方針:

- 業務テーブルとして密度高め
- 重要列を左に固定する
- 備考は折り返し表示し、長文は展開式
- キャンセルは破壊的操作なので確認を挟む

### 7.4 ドライバー画面

目的:

次の訪問先と次の操作だけを迷わず判断できる。

主要要素:

- 担当車両
- AM/PM
- 残り人数
- 次の訪問先
- 住所
- 電話
- 備考
- 到着、乗車/降車、出発、完了
- 直前取消

レイアウト:

```text
┌────────────────────┐
│ 車両A AM  残り3名     │
├────────────────────┤
│ 次の訪問先             │
│ 09:20  利用者名         │
│ 住所                   │
│ 備考                   │
├────────────────────┤
│ [到着]                 │
│ [乗車/降車]             │
│ [出発]                 │
│ [完了]                 │
├────────────────────┤
│ [電話] [ナビ] [直前取消] │
└────────────────────┘
```

UI方針:

- ボタンは縦積み、大きくする
- その時点で押せないボタンは無効化し、理由を表示する
- 完了済みの人を次の訪問先に出さない
- 電話・ナビは初回デモではリンク風UIまでにする
- 「運転中は操作しない」注意を常時表示するが、説明文で画面を埋めない

### 7.5 進捗モニター

目的:

施設職員が「いま何が起きているか」を確認する。

主要要素:

- 車両別ステータス
- 現在対応中の利用者
- 完了人数
- 残人数
- 最終更新
- 遅延候補

UI方針:

- 編集ボタンを置かない
- 問い合わせに必要な最小情報を表示する
- 個人情報はマスク表示を初期値にできるようにする

### 7.6 実績画面

目的:

送迎後に予定と実績の差分を確認し、CSV出力する。

主要要素:

- 車両フィルタ
- AM/PMフィルタ
- 予定時刻
- 到着時刻
- 乗車/降車時刻
- 出発時刻
- 完了時刻
- 遅延時間
- CSV出力

UI方針:

- 予定と実績を横並びにする
- 遅延は分単位で表示する
- CSV出力前に件数と対象日を表示する

## 8. ビジュアルデザイン

### デザイン方向

落ち着いた運行管理UI。SaaSの派手なグラデーションや装飾ではなく、現場で毎日使える密度と明瞭さを優先する。

避けるもの:

- ランディングページ風のヒーロー
- 大きな装飾カード
- 紫や青のグラデーション主体
- 意味のないイラスト
- 色だけの状態表現

### カラートークン

```text
--color-paper: #F7F7F2      // 背景。紙の送迎表に近い温度
--color-ink: #202124        // 本文
--color-muted: #5F6368      // 補助テキスト
--color-line: #D7D4CA       // 区切り線
--color-lane: #265C4B       // 送迎レーンの主色
--color-action: #1E6C9E     // 主要操作
--color-warning: #9A5A00    // 遅延・注意
--color-danger: #B3261E     // キャンセル・エラー
--color-complete: #2E7D32   // 完了
--color-surface: #FFFFFF    // パネル
```

理由:

- 紙の送迎表からWebへ移行するため、背景は白すぎない
- 主色は介護・送迎の落ち着きと可読性を重視した緑
- 操作色は主色と分離し、押せるものを認識しやすくする
- 警告・危険・完了は色相を分ける

### タイポグラフィ

推奨:

- UI/本文: `Noto Sans JP`
- 数字・時刻・車両番号: `Roboto Mono` または `ui-monospace`

方針:

- 日本語本文は読みやすさ優先
- 時刻と件数は等幅で桁ズレを防ぐ
- 見出しは大きくしすぎない

サイズ:

```text
page title: 24px / 32px / 700
section title: 18px / 28px / 700
body: 15px / 24px / 400
table: 14px / 22px / 400
caption: 12px / 18px / 500
driver primary: 22px / 32px / 700
button: 16px / 24px / 700
```

### 角丸・余白

- パネル角丸: 8px
- 小ボタン角丸: 6px
- 大ボタン角丸: 8px
- ページ左右余白: desktop 24px, mobile 16px
- 表セル余白: 8px 10px
- ドライバーボタン高さ: 56px以上

## 11. 操作設計

### ボタン優先順位

Primary:

- 取込
- 到着
- 乗車/降車
- 出発
- 完了
- CSV出力

Secondary:

- AM/PM切替
- 車両切替
- ロール切替
- 電話
- ナビ

Danger:

- キャンセル
- 取込データ破棄

### 破壊的操作

キャンセル操作:

- 確認を挟む
- 理由入力を任意または必須にする
- 実績画面に残す

取込やり直し:

- 現在の操作履歴が消える場合は確認する

### エラー文言

悪い例:

- エラーが発生しました
- 失敗しました

良い例:

- `Sheet1 H5 の予定時刻を読み取れません。Excelの時刻形式を確認してください。`
- `車両名が空です。Sheet1の車両欄を確認してください。`

## 13. アクセシビリティ要件

### 色とコントラスト

- テキストは背景に対して4.5:1以上を目標
- 状態表現は色、ラベル、形を併用
- 強制カラーモードでも意味が失われないようにする

### キーボード

- すべてのボタンにTabで到達できる
- フォーカスリングを消さない
- モーダルはEscで閉じられる
- モーダル内でフォーカスが迷子にならない

### 読み上げ

- アイコンボタンにはaria-labelを付ける
- 状態変更はテキストでも表示する
- テーブルの見出しを正しく設定する

### 動き

- 自動で流れるアニメーションは使わない
- 進捗更新は点滅させない
- `prefers-reduced-motion` を尊重する

## 14. 個人情報表示設計

### 通常表示

管理者とドライバーは業務上必要な範囲で表示する。

- 氏名
- 電話
- 住所
- 備考

### デモ表示

画面共有や外部説明ではマスク可能にする。

## 15. UXライティング

### 基本方針

- 現場語で書く
- 操作結果を明確にする
- システム都合の言葉を避ける

ラベル例:

| 機能 | ラベル |
| --- | --- |
| Excel取込 | 送迎表を取り込む |
| Import success | 取込が完了しました |
| Cancel | キャンセルにする |
| Undo | 直前の操作を戻す |
| Export | 実績CSVを出力 |
| Error | 確認が必要です |

## 16. Devil's Advocateレビュー

### Critical

- ドライバー画面が便利すぎると、運転中操作を誘発する可能性がある。初回デモでも、操作は停車中前提の文言と大きなボタンに限定する。
- 送迎安全の文脈では、完了操作だけでは不十分である。将来は乗車・降車・車内確認の明示的チェックが必要。
- 実Excelには個人情報がある。UI設計でマスク表示を作っても、ログやテスト出力で漏れれば意味がない。

### Warning

- 地図なしのデモは、らくぴた送迎の印象と差が出る。ただし初回はExcel見える化と実績記録を優先する。
- 車両レーンUIは見やすい一方、利用者が多い場合に横幅が足りない。Desktopは横、Mobileは縦へ切り替える。
- キャンセル操作と完了操作が近いと誤操作につながる。Danger操作は位置と色を分ける。

### Suggestion

- 初回デモでは「現場で使える最小UI」を見せ、本番提案では地図、GPS、通知、安全確認強化を追加提案にする。
- UIの独自性は装飾ではなく、送迎レーンという業務構造で出す。

### Verdict

CONDITIONAL APPROVE.

条件:

- 初回実装ではドライバー画面を最優先で実機幅確認する
- 個人情報マスクをUI要件に含める
- 状態表現を色だけにしない
- キャンセルと完了の誤操作対策を入れる
